# 성도관리 매뉴얼 기반 이음앱 점검

기준 문서: `01. Ministry/01. 이음채플/04. Formation & Care/성도관리 매뉴얼.md`

## 1. 점검 요약

이음앱은 새가족, 멤버, 순, 출석, 사역팀 기능을 이미 갖추고 있어 성도관리 매뉴얼을 운영할 기본 토대는 있다.

다만 현재 앱의 상태 체계는 `new_family -> adjusting -> attending` 중심이고, 매뉴얼의 핵심인 `방문자 -> 새가족 -> 성도 - 연결 진행 중 -> 성도 - 연결 완료` 흐름과 완전히 일치하지 않는다.

가장 큰 차이는 새가족교육 완료 이후의 처리다. 현재 앱은 3주차 교육 완료 시 자동으로 `adjusting` 상태가 되고, 3개월이 지나면 자동으로 `attending` 상태가 된다. 매뉴얼 기준에서는 교육 완료만으로 끝나는 것이 아니라, 본인 소속 의사와 목양/순/팀 연결 확인이 함께 필요하다.

## 2. 잘 맞는 부분

### 2.1 새가족 전용 관리 화면이 있다

`/new-family` 화면에서 새가족을 등록하고, 담당자를 지정하고, 단계별 진행 상태를 볼 수 있다.

현재 단계:
- 1주차 방문
- 2주차 교육
- 3주차 교육

이는 매뉴얼의 "새가족교육 전/진행 중" 관리 흐름과 잘 맞는다.

### 2.2 담당자 지정이 가능하다

`new_family.assigned_to`가 있고, 화면에서 담당자를 선택할 수 있다. 이는 매뉴얼의 "목양 담당자 또는 연결자 지정" 기준과 맞는다.

### 2.3 순 연결 상태를 볼 수 있다

멤버 목록에서 현재 시즌 기준 순 배정 정보를 함께 보여준다. 이는 매뉴얼의 "연결 완료" 판단 요소 중 하나다.

### 2.4 장기 미진행 감지가 있다

새가족 화면에서 2주 이상 단계 변화가 없는 사람을 경고하고, 공통 쿼리에는 4주 이상 정체된 새가족을 이탈 처리하는 함수가 있다.

이 방향은 매뉴얼의 "정기 점검"과 "돌봄 필요" 흐름에 사용할 수 있다.

## 3. 매뉴얼과 어긋나는 부분

### 3.1 방문자와 새가족이 앱에서 분리되지 않는다

방문자 카드 제출 시 `members.status`가 바로 `new_family`로 저장되고, `new_family` 레코드도 즉시 생성된다.

현재 흐름:
- 방문자 카드 제출
- members에 `new_family`로 등록
- new_family에도 등록

매뉴얼 기준:
- 방문자는 아직 소속 의사가 확인되지 않은 초기 방문자
- 반복 출석 또는 소속 관심이 확인되면 새가족

따라서 현재 앱은 "첫 방문자"와 "정착 가능성이 확인된 새가족"을 같은 상태로 취급한다.

영향:
- 새가족교육 대상자와 단순 방문자를 구분하기 어렵다.
- 새가족 후속조치 우선순위가 흐려질 수 있다.
- 방문자가 곧바로 멤버 테이블에 들어가므로 현원 통계 해석이 애매해질 수 있다.

### 3.2 교육 완료가 자동으로 성도 전환처럼 작동한다

`updateStep()`에서 3주차 교육 완료 시 `members.status`를 `new_family`에서 `adjusting`으로 자동 변경한다.

매뉴얼 기준:
- 새가족교육 완료
- 이음채플 소속 의사 확인
- 기본 정보 확인
- 목양 담당자, 순, 팀 중 최소 하나 연결 확인

현재 앱은 교육 완료 이후 "소속 의사 확인"과 "연결 확인"을 별도 데이터로 요구하지 않는다.

영향:
- 교육은 마쳤지만 연결이 안 된 사람도 자동으로 성도 흐름에 들어간다.
- "성도 - 연결 진행 중"과 "성도 - 연결 완료"를 구분할 수 없다.

### 3.3 `adjusting` 상태가 매뉴얼의 "연결 진행 중"과 정확히 같지 않다

현재 `adjusting`은 화면에서 "적응중"으로 표시되고, 멤버 메인 상태로는 "재적"에 포함된다.

매뉴얼 기준으로는 이 상태를 "성도 - 연결 진행 중"에 가깝게 사용할 수 있지만, 현재 앱에서는 다음 정보가 없다.

- 소속 의사 확인 여부
- 연결 상태: 미연결 / 진행 중 / 완료
- 목양 담당자
- 연결 지연 사유

따라서 `adjusting`만으로는 매뉴얼의 연결 진행 상태를 충분히 표현하지 못한다.

### 3.4 3개월 후 자동 `attending` 전환은 목양 판단을 건너뛴다

`expireAdjustingMembers()`는 3주차 교육 완료 후 3개월이 지난 `adjusting` 멤버를 자동으로 `attending`으로 바꾼다.

매뉴얼 기준:
- 4주 내 연결 상태 확인
- 8주 내 실제 관계 형성 여부 확인
- 필요 시 추가 돌봄 또는 재연결

현재 앱 기준:
- 시간이 지나면 자동으로 출석 멤버가 된다.

영향:
- 순/목양/팀 연결이 없어도 자동으로 완료된 것처럼 보일 수 있다.
- 목양적 확인 없이 통계가 좋아 보이는 방향으로 정리될 수 있다.

### 3.5 이탈 처리가 "돌봄 필요"보다 "비활성화"에 가깝다

`markDroppedOutNewFamilies()`는 4주 이상 정체된 새가족을 `dropped_out = true`로 표시하고, 멤버 상태를 `inactive`으로 바꾼다.

매뉴얼 기준:
- 4주 이상 미출석 시 안부 연락
- 8주 이상 지속 시 목양 담당자 또는 리더 보고
- 12주 이상 지속 시 상태 재분류 검토

현재 앱 기준:
- 4주 정체 시 이탈/미출석 처리

영향:
- 목양적 확인 전 "이탈"로 너무 빨리 분류될 수 있다.
- "돌봄 필요" 상태가 앱에 없어서 안부 연락과 재분류 사이의 중간 지점이 부족하다.

### 3.6 상태명이 운영 언어와 다르다

현재 앱 상태:
- 재적
- 출석
- 미출석
- 제적
- 휴적
- 새가족
- 적응중

매뉴얼 상태:
- 방문자
- 새가족
- 성도 - 연결 진행 중
- 성도 - 연결 완료
- 돌봄 필요
- 장기 미출석
- 이동/종료

운영자가 앱을 볼 때 `재적`, `출석`, `적응중`이 각각 매뉴얼의 어느 상태인지 바로 알기 어렵다.

## 4. 추천 상태 매핑

현재 DB 상태를 당장 크게 바꾸지 않는다면, 우선 아래처럼 해석한다.

| 앱 상태 | 매뉴얼상 의미 | 비고 |
| --- | --- | --- |
| new_family | 새가족 | 단, 첫 방문자는 별도 표시 필요 |
| adjusting | 성도 - 연결 진행 중 | 교육 완료 후 연결 확인 전 |
| attending | 성도 - 연결 완료 | 연결 확인 후에만 전환 권장 |
| active | 성도 - 연결 완료 또는 재적 성도 | 기존 데이터 정리 필요 |
| inactive | 장기 미출석 | 돌봄 필요와 분리 필요 |
| on_leave | 휴적 | 매뉴얼의 특수 상태 |
| removed | 이동/종료 | 제적보다 이동/종료 사유 권장 |

## 5. 개선 우선순위

### P1. 자동 성도 전환을 멈추고 확인 단계를 추가한다

3주차 교육 완료 시 바로 `attending` 또는 완료 성격의 상태로 가지 않고, `adjusting`을 "성도 - 연결 진행 중"으로 명확히 사용한다.

추가로 필요한 확인 항목:
- 소속 의사 확인
- 연결 상태
- 목양 담당자
- 순/그룹
- 다음 조치

### P1. 3개월 자동 `attending` 전환을 제거하거나 수동 승인으로 바꾼다

`adjusting -> attending`은 시간이 아니라 연결 확인으로 전환되어야 한다.

추천:
- 자동 전환 대신 "연결 확인 필요" 목록에 표시
- 관리자가 "연결 완료" 버튼을 누르면 `attending`으로 변경

### P1. 방문자와 새가족을 분리한다

방문자 카드 제출자는 처음에는 `visitor` 또는 별도 `visitor_cards` 상태로 보관하고, 재방문/소속 관심 확인 후 새가족으로 전환하는 흐름이 좋다.

단기 대안:
- DB 상태는 `new_family`를 유지하되, `new_family`에 `interest_confirmed` 또는 `visitor_stage` 필드를 추가한다.

장기 대안:
- `visitor_cards` 테이블을 만들고, 관리자가 "새가족 전환"을 누르면 `members`와 `new_family`로 생성한다.

### P2. "돌봄 필요" 상태를 추가한다

4주 이상 정체되었다고 바로 `inactive`으로 보내기보다, 먼저 `care_needed` 또는 `needs_follow_up` 상태를 두는 것이 매뉴얼과 맞다.

추천:
- 2주 정체: 화면 경고
- 4주 정체: 돌봄 필요
- 8주 정체: 리더 보고
- 12주 정체: 장기 미출석 또는 이동/종료 검토

### P2. 연결 상태를 별도 필드로 둔다

멤버의 정체성 상태와 연결 상태를 분리한다.

추천 필드:
- `connection_status`: `not_started`, `in_progress`, `completed`
- `care_owner_id`: 목양 담당자
- `connection_note`: 연결 지연 사유 또는 다음 조치
- `membership_confirmed_at`: 소속 의사 확인일
- `new_family_completed_at`: 새가족교육 완료일

### P2. 멤버 화면 필터명을 매뉴얼 언어에 맞춘다

`적응중` 대신 "연결 진행 중" 또는 "정착 중"처럼 운영자가 바로 이해할 수 있는 표현이 좋다.

추천 표시:
- `new_family`: 새가족
- `adjusting`: 연결 진행 중
- `attending`: 성도
- `inactive`: 장기 미출석

## 6. 구현 제안

### 1단계: 용어와 자동화 정리

- `adjusting` 표시명을 "연결 진행 중"으로 변경
- 3개월 자동 `attending` 전환 제거
- 새가족 완료 안내 문구를 "연결 진행 중으로 변경됩니다"로 수정

### 2단계: 연결 확인 필드 추가

- `members` 또는 별도 `member_care` 테이블에 연결 상태 필드 추가
- 새가족 완료자 목록에 "연결 확인 필요" 섹션 추가
- 순/담당자/사역팀 중 연결 요소를 한눈에 표시

### 3단계: 방문자 분리

- 방문자 카드를 바로 새가족으로 넣지 않고 방문자 목록에 저장
- 관리자가 새가족으로 전환하는 액션 추가
- 재방문/소속 관심 확인 상태 관리

## 7. 결론

이음앱은 이미 성도관리 매뉴얼을 얹을 수 있는 뼈대가 있다. 특히 새가족 단계, 담당자, 순, 출석, 사역팀 연결은 좋은 기반이다.

다만 현재 앱은 사람의 상태를 "교육 단계와 시간"으로 자동 변경하는 경향이 있다. 새 매뉴얼 기준으로는 "소속 의사 확인"과 "공동체 연결 확인"이 전환의 핵심이 되어야 한다.

우선순위는 다음 순서가 좋다.

1. `adjusting`을 "성도 - 연결 진행 중"으로 재정의한다.
2. 3개월 자동 `attending` 전환을 수동 연결 완료로 바꾼다.
3. 방문자와 새가족을 분리한다.
4. 돌봄 필요/장기 미출석 흐름을 만든다.
